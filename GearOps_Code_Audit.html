Security

3 critical · 2 high · includes secrets committed to Git

▾
Critical
Firebase service account private key committed to Git — in two places
assetmanager-dev-3a7cf-firebase-adminsdk-fbsvc-221bff9e42.json (root) · config/assetmanager-dev-3a7cf-firebase-adminsdk-fbsvc-221bff9e42.json

Your Firebase Admin SDK service account JSON — which contains a private RSA key — is committed to source control in two separate locations. Anyone with repo access can impersonate any Firebase user, create custom tokens, read/write your entire database, bypass all auth, and delete accounts. This is the highest-severity secret exposure possible in a Firebase project.

Fix immediately:
  • → Go to Firebase Console → Project Settings → Service Accounts → Revoke current key NOW
  • → Generate a new service account key, store it ONLY as an environment variable or AWS Secrets Manager
  • → Add both file paths to .gitignore immediately
  • → If repo is or was ever public, rotate the key even if you never pushed it publicly — Git history is permanent
  • → In production: use GOOGLE_APPLICATION_CREDENTIALS env var pointing to the file path on EC2, not a committed JSON
Critical
Auth middleware bypass: any request can skip Firebase token verification
inventory-api/middleware/auth.js · lines 52–63

The dev bypass uses process.env.NODE_ENV || 'development'. If NODE_ENV is ever unset on EC2 (missing in PM2 config, a reboot, a bad deploy), the entire auth layer silently disables itself. Any HTTP request with an X-User-Id header becomes authenticated as any user ID.

// auth.js line 52 — THE BUG
if ((process.env.NODE_ENV || 'development') !== 'production') {
  // This runs when NODE_ENV is undefined — i.e. on a misconfigured production server
  const uidFromHeader = req.header('X-User-Id');
  if (uidFromHeader) {
    req.user = { uid: String(uidFromHeader) }; // bypasses Firebase entirely
    return next();
  }
}
Fix: Remove the || 'development' fallback. Explicitly check process.env.NODE_ENV === 'development'. Add a startup assertion that crashes the server if NODE_ENV is missing in production. Add NODE_ENV=production to your PM2 ecosystem config.
Critical
All assets fetched without authentication
utils/fetchTaskCount.js · line 45 · inventory-api/routes/assets.js

The GET /assets endpoint is called without any Bearer token in fetchTaskCount.js, and also in the same utility used for the tab badge count. If this route is publicly accessible (no auth middleware), every asset in your entire database is readable without logging in. Verify that the assets route has authRequired middleware applied.

Fix: Confirm GET /assets in inventory-api/routes/assets.js uses authRequired middleware. Pass the Firebase ID token in fetchTaskCount.js using getAuth().currentUser.getIdToken() before making the call.
High
AWS credentials as environment variables instead of IAM role
inventory-api/routes/assets.js · lines 40–44

Using explicit AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY means those long-lived credentials can leak through logs, error dumps, or a compromised .env file. EC2 should use an IAM instance role — credentials are then temporary, auto-rotated, and never stored anywhere.

Fix: In AWS Console → IAM → Create role for EC2 → attach an S3 policy scoped to your bucket → Attach the role to your EC2 instance. Remove the explicit keys from your S3 constructor — the SDK automatically picks up instance role credentials. Delete the env vars from your .env file.
High
No rate limiting on any endpoint
inventory-api/ — no rate limiting middleware found

Your API has no rate limiting. Auth endpoints are fully open to brute-force attacks. File upload endpoints (POST /assets) can be hammered to exhaust S3 storage and EC2 CPU. QR generation endpoints can be abused to create thousands of documents.

Fix: Add express-rate-limit. Apply a strict limit (5 req/15min) on auth-adjacent routes, a moderate limit (100 req/min) on data routes, and a very strict limit (3 req/min) on file upload routes. 
const rateLimit = require('express-rate-limit');
app.use('/api/', rateLimit({ windowMs: 60_000, max: 100 }));
app.use('/api/assets', rateLimit({ windowMs: 60_000, max: 30 })); // uploads
Medium
No input sanitisation on custom asset fields stored as JSON
inventory-api/routes/assets.js · custom field storage

Custom field values arrive from the client and are stored in the database as JSON without sanitisation. If any field value is rendered as HTML in the web dashboard (directly set as innerHTML or passed unsafely to a component), this is a stored XSS vector. Even without XSS today, validating field shapes prevents data corruption.

Fix: Sanitise string values with a library like DOMPurify (frontend) or xss (backend) before storage. Enforce field type validation against the asset type's field schema before inserting.

Architecture — Frontend Doing Backend Work

Business logic that must move to the API

▾
High
Entire task-count algorithm runs on every client device
utils/fetchTaskCount.js · 132 lines of business logic on the frontend

This file downloads all assets, then for each unique asset type makes a separate API call to get field definitions, then loops through every asset and field to calculate due dates and lead days. This is O(assets × types) in network requests, all running on the user's phone. With 500 assets it means ~50 API calls just to show a badge number. This logic belongs 100% on the server.

Fix: Create GET /assets/tasks/count?userId=X endpoint on the backend. The server has direct DB access — it can compute this in a single SQL query with JOINs. The frontend just fetches one number. Delete fetchTaskCount.js entirely. 
-- Single RDS query replaces 132 lines of frontend JS + N API calls
SELECT COUNT(*) FROM asset_field_values afv
JOIN assets a ON a.id = afv.asset_id
WHERE afv.value_type = 'date'
  AND CAST(afv.value AS DATE) < CURRENT_DATE
  AND (a.assigned_to_id = $1 OR $2 = true);
High
Permission / role checks scattered across 8+ frontend files
dashboard.js, check-in/[id].js, admin/index.js, index.js, tasks.js…

Admin checks use three different approaches: Firebase custom claim (tokenResult.claims.admin), database role lookup (GET /users/:id → check role === 'ADMIN'), and email heuristic (email.includes('admin@')) in check-in. These get out of sync. The email heuristic in particular is a security bug — anyone with "admin" in their email gets elevated UI access.

Fix: Create a single useUserRole() hook that returns { isAdmin, role, canX }. The hook fetches once on mount and stores in context. Remove all inline permission checks from components. Delete the email-pattern check entirely — it should never exist.
High
15 separate PrismaClient instances — database connection pool exhaustion
assets.js, users.js, auth.js, assetTypes.js, hireDisclaimer.js, assetDocuments.js, activity.js, controllers/*, services/*…

Every file that needs the database instantiates its own new PrismaClient(). Each Prisma client opens its own connection pool to RDS. With 15 instances and a default pool size of 10, you're potentially opening 150 database connections under load. RDS connection limits are finite. This will cause connection exhaustion and crashes under real traffic.

// Found in 15 separate files:
const prisma = new PrismaClient(); // ← each one opens its own pool
Fix: Create inventory-api/lib/prisma.js with a singleton pattern. Every file imports from this one module. 
// inventory-api/lib/prisma.js
const { PrismaClient } = require('../generated/prisma');
const globalForPrisma = global;
const prisma = globalForPrisma.prisma ?? new PrismaClient();
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma;
module.exports = prisma;

// Every other file becomes:
const prisma = require('../lib/prisma'); // shared singleton
Medium
No API client layer — raw fetch() calls scattered across 22 frontend files
86 await .json() calls across the app with no centralised error handling

There is no API client abstraction. Every screen constructs its own fetch URL, adds its own headers, and handles (or ignores) its own errors. If the base URL changes, the auth header changes, or you want to add logging, you need to update 22 files. Additionally, 86 .json() calls were found but only ~108 response.ok checks across 22 files — meaning many failed API calls (500, 404, 403) are silently parsed as if they succeeded.

Fix: Create utils/apiClient.js — a thin wrapper around fetch that handles auth headers, checks response.ok, throws on errors, and adds a request ID for tracing. 
// utils/apiClient.js
import { getAuth } from 'firebase/auth';
import { API_BASE_URL } from '../inventory-api/apiBase';

async function apiFetch(path, options = {}) {
  const token = await getAuth().currentUser?.getIdToken();
  const res = await fetch(`${API_BASE_URL}${path}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...options.headers,
    },
  });
  if (!res.ok) {
    const err = await res.json().catch(() => ({}));
    throw new Error(err.error || `API ${res.status}: ${path}`);
  }
  return res.json();
}
export const api = {
  get: (path) => apiFetch(path),
  post: (path, body) => apiFetch(path, { method: 'POST', body: JSON.stringify(body) }),
  patch: (path, body) => apiFetch(path, { method: 'PATCH', body: JSON.stringify(body) }),
  del: (path) => apiFetch(path, { method: 'DELETE' }),
};

Performance

N+1 queries, no pagination, connection pool exhaustion

▾
High
N+1 query pattern — one API call per asset type on every tab badge refresh
utils/fetchTaskCount.js · lines 65–73

After fetching all assets, the code extracts unique type IDs then loops over them, making an individual fetch per type to get field definitions. With 10 unique asset types this is 11 network requests (1 for all assets + 10 for types). This runs every time the tab bar badge needs to update.

// N+1 pattern — one fetch per unique type ID
for (const tId of typeIds) {
  const r = await fetch(`${API_BASE_URL}/assets/asset-types/${tId}/fields`);
  // runs sequentially, blocks UI thread
}
Fix: Move to a single backend endpoint as described in the Architecture section. If you keep the frontend approach temporarily, at least batch: GET /asset-types/fields/batch?ids=1,2,3 or use Promise.all() to run in parallel instead of sequentially.
Medium
No pagination — full asset list loaded for every task count refresh
utils/fetchTaskCount.js · line 45 · inventory-api/routes/assets.js

The task count utility fetches GET /assets with no limit or offset. As the fleet grows to 500+ assets, this returns the full dataset on every badge refresh, on every device, every time the app is foregrounded. The backend route should support ?limit=&offset= parameters, and the task count logic should move server-side entirely.

Fix (short-term): Add ?limit=500&fields=id,type_id,assigned_to_id,fields to reduce payload size. Fix (proper): Create the server-side task count endpoint.
Medium
Response body consumed twice — double .json() parse
utils/fetchTaskCount.js · line 48

The response body stream is read twice on the same response object. The first await assetsRes.json() consumes the stream; the second call will always return an empty result or throw.

// BUG: consumes body stream twice
const list = Array.isArray(await assetsRes.json())
  ? await assetsRes.json() // ← stream already consumed, returns nothing
  : [];
Fix: const raw = await assetsRes.json(); const list = Array.isArray(raw) ? raw : [];
Medium
Massive single-file components causing unnecessary full re-renders
tasks.js (2,334 lines) · check-in/[id].js (2,240 lines) · asset/[assetId].js (2,168 lines)

These files are monolithic components with 30+ state variables each. Any state change triggers a re-render of the entire component tree. Modals, lists, forms and data all live in one component, so showing a loading spinner causes the whole 2,000-line component to re-evaluate.

Fix: Split into sub-components. For example, check-in/[id].js should be broken into AssetHeader, UserSelectModal, SwapModal, ActionsPanel. Use React.memo on list items and modals that don't need to re-render with parent state changes.
Medium
No image caching strategy for S3 asset photos
app/asset/[assetId].js · all screens rendering asset images

Asset images are loaded directly from S3 URLs on every render with no local cache. On a slow connection, scrolling a list of assets causes network requests for every visible image on every scroll. React Native's default Image component does minimal caching.

Fix: Install expo-image (replaces the built-in Image) which has aggressive disk + memory caching. For lists, use FlashList from Shopify instead of FlatList for better recycling performance.

Code Quality

83 console.log calls · dead code · inconsistent patterns

▾
Medium
83 console.log/warn/error calls across 18 files — all leak to production
check-in/[id].js (27 occurrences) · admin/index.js (17) · tasks.js (7)…

Every debug log you wrote during development ships to every user's device in production. These logs can expose user IDs, token lengths, API responses, and internal state. They also clutter the console for legitimate error monitoring.

Fix: Create a utils/logger.js that wraps console and no-ops in production. Replace all direct console calls. 
// utils/logger.js
const isDev = process.env.NODE_ENV !== 'production';
export const logger = {
  log:   (...a) => isDev && console.log(...a),
  warn:  (...a) => isDev && console.warn(...a),
  error: (...a) => console.error(...a), // errors always log
};
Medium
Status values hardcoded as magic strings in multiple files
asset/edit.js, Inventory.js, check-in/[id].js, tasks.js…

Asset status strings like 'In Service', 'On Hire', 'Maintenance' appear as raw string literals in multiple files. If you rename a status value, you have to find and replace it everywhere — and you will miss one.

Fix: Add to constants/index.js: 
export const ASSET_STATUS = {
  IN_SERVICE:  'In Service',
  ON_HIRE:     'On Hire',
  MAINTENANCE: 'Maintenance',
  REPAIR:      'Repair',
  EOL:         'End of Life',
};
Medium
TypeScript config exists but the entire codebase is plain JavaScript
tsconfig.json present · all app files are .js

You've got a tsconfig.json but haven't migrated to TypeScript. This means API response shapes are untyped — any field can be accessed without IDE error, typos in field names are invisible, and refactoring is risky. Given that your API is now well-defined with Prisma (which generates types), this is low-hanging fruit.

Fix: Start incrementally. Rename new files to .ts/.tsx. Use Prisma's generated types for all API responses. Start with the API client layer and hooks — they provide the most leverage.
Low
Empty catch blocks silently swallow errors
tasks.js, fetchTaskCount.js, various async functions

Several catch blocks contain only a comment or are completely empty. When these fail in production, there is no trace of what went wrong, no user feedback, and no way to debug it.

Fix: At minimum, every catch block should call logger.error(). Better: integrate Sentry (@sentry/react-native) for automatic error capture with stack traces.

Input Validation

Good foundations — specific gaps to close

▾

What's already good

Email domain allowlist on registration
Required field validation on asset create
Document requirement enforcement
File type/size limits on upload middleware
High
API endpoints do not re-validate inputs server-side
inventory-api/routes/assets.js · users.js · all POST/PATCH routes

Frontend validation is good — but it only runs when the user submits the UI. Anyone using curl, Postman, or a script can send any payload directly to your API, bypassing all client-side checks. Fields like assigned_to_id, status, custom field values, and user roles are never validated on the server before hitting the database.

Fix: Add zod or joi schema validation middleware to all POST/PATCH routes. Validate shape, types, enum values, and string lengths before the controller runs. Return 400 with a descriptive error if validation fails.
Medium
Numeric fields (cost, rates) handled as strings — no type coercion or range validation
components/ActionsForm.js · asset/new.js

Cost and rate fields are read from TextInput as strings and sent directly to the API. A user typing "abc" or "-9999" will store invalid data. The database will accept whatever string you send.

Fix: Use keyboardType="numeric" on currency inputs. Validate with !isNaN(parseFloat(value)) && parseFloat(value) >= 0 before submission. Coerce to float on the server before inserting.
Medium
No maximum length validation on text fields
app/asset/new.js · app/type/new.js · all TextInput fields

Free-text fields like asset name, description, and notes have no maxLength prop and no server-side length check. A user could submit a 100,000-character string which will be stored in your RDS database and returned in every API response that includes that asset.

Fix: Add maxLength props to all TextInput components. Add corresponding max: N constraints to your Zod/Joi schemas. Add @db.VarChar(255) or similar constraints to your Prisma schema.
Low
Date fields accept free text — no format validation before API submission
Dynamic custom fields in asset/new.js and type/edit.js

Date-type custom fields are entered as text strings. If a user types "tomorrow" or "01/04/26" instead of "2026-04-01", the value is stored as-is. The isDateLike() function in fetchTaskCount.js then tries to parse arbitrary strings, which works sometimes but silently fails others.

Fix: Use a date picker component (@react-native-community/datetimepicker) for all date fields. Store ISO 8601 format YYYY-MM-DD only. Reject non-ISO strings at the API level.

Refactor
Shared hook pattern — useAssetTypeFields(typeId)

Asset type field definitions are fetched individually by at least 5 screens. A cached hook would eliminate the duplicate network calls and ensure all screens see the same field definitions without out-of-sync state.

// hooks/useAssetTypeFields.js
const cache = new Map();

export function useAssetTypeFields(typeId) {
  const [fields, setFields] = useState(cache.get(typeId) || []);
  const [loading, setLoading] = useState(!cache.has(typeId));

  useEffect(() => {
    if (!typeId || cache.has(typeId)) return;
    api.get(`/assets/asset-types/${typeId}/fields`)
      .then(data => { cache.set(typeId, data); setFields(data); })
      .finally(() => setLoading(false));
  }, [typeId]);

  return { fields, loading };
}
